## Webhook Security (T-65, T-67)

🔴 A webhook with a valid HMAC signature computed against the raw body bytes is accepted and processed
  Scenario    : Send a POST to /api/payment/webhook with a Razorpay-formatted payload and a correct HMAC-SHA256 signature (computed against the exact raw byte sequence of the body, using RAZORPAY_WEBHOOK_SECRET). Use the subscription.activated event.
  Pass if     : Server returns 200 and the subscription row in the DB transitions to the expected state.
  Fail if     : Server returns 4xx, or the DB row is not updated, or the signature check is bypassed.
  Automatable : yes
  @critical

🔴 A webhook with a signature computed against re-serialized JSON (not raw bytes) is rejected
  Scenario    : Take a valid webhook payload, re-serialize the body through JSON.parse/JSON.stringify (which can reorder keys or strip whitespace), compute HMAC against that re-serialized string, and POST to /api/payment/webhook.
  Pass if     : Server returns 400 or 401 — signature mismatch detected.
  Fail if     : Server returns 200 and processes the event, meaning the implementation re-serializes before verification (critical security defect — renders signature verification worthless).
  Automatable : yes
  @critical

🔴 A webhook with a tampered body (signature does not match) is rejected with no DB side-effects
  Scenario    : POST to /api/payment/webhook with a syntactically valid JSON payload but an incorrect HMAC signature header.
  Pass if     : Server returns 400 or 401 immediately; no row is written or updated in subscriptions or processed_webhook_events.
  Fail if     : Server returns 200 or any DB write occurs.
  Automatable : yes
  @critical

🔴 Duplicate webhook event delivery does not double-grant or double-extend access (idempotency)
  Scenario    : Send the same webhook event (same razorpay_event_id) twice in sequence. First delivery should be processed normally. Second delivery should be silently acknowledged.
  Pass if     : Server returns 200 for both calls. The subscription row reflects the state from the first delivery only. processed_webhook_events contains exactly one row for that event ID.
  Fail if     : A second DB write occurs on the duplicate (extending current_period_end, re-activating, etc.), or the server returns an error on the second call.
  Automatable : yes
  @critical

🔴 A late out-of-order webhook event does not overwrite a more recent subscription state
  Scenario    : Simulate the DB already holding a subscription.charged event result (status=active, current_period_end=future). Then deliver a subscription.halted event with an older event timestamp to /api/payment/webhook.
  Pass if     : The subscription row retains status=active; the late halted event is discarded or logged without writing. Server returns 200.
  Fail if     : status is overwritten to halted or current_period_end is modified, representing a timestamp-guard failure.
  Automatable : yes
  @critical

## Access Gate Middleware (T-68)

🔴 A request to a protected /api/* route is blocked (402) when PAYMENT_ENABLED=true and no active subscription row exists
  Scenario    : Start server with PAYMENT_ENABLED=true (RAZORPAY_KEY_ID present). Make GET /api/signals (or any non-payment API route) with no subscription row in the DB.
  Pass if     : Response is HTTP 402 with body {"error":"subscription_required","redirectTo":"/pricing"}.
  Fail if     : Request succeeds (200), or any other non-402 status is returned.
  Automatable : yes
  @critical

🔴 A request to /api/payment/* is allowed through the access gate even when PAYMENT_ENABLED=true and no subscription exists
  Scenario    : Start server with PAYMENT_ENABLED=true. Make GET /api/payment/plans and POST /api/payment/create-subscription with no active subscription row.
  Pass if     : Both endpoints respond (200 or 422/validation error based on input) — they are never 402-blocked.
  Fail if     : Either payment endpoint returns 402, locking users out of the ability to subscribe.
  Automatable : yes
  @critical

🔴 All /api/* routes pass through without subscription check when PAYMENT_ENABLED=false (no payment keys configured)
  Scenario    : Start server with RAZORPAY_KEY_ID absent (PAYMENT_ENABLED effectively false). Make requests to protected API routes.
  Pass if     : All routes respond normally; no 402 is ever returned; access gate is fully transparent.
  Fail if     : Any 402 is returned, or the server errors on startup due to missing payment env vars.
  Automatable : yes
  @critical

🔴 A subscription in grace period (current_period_end between 3 days ago and now) still grants access
  Scenario    : Insert a subscription row with status=active and current_period_end = now() - 2 days (within the 3-day grace window). Make a request to a protected route.
  Pass if     : Request succeeds (200) — grace period is respected.
  Fail if     : Request is blocked with 402.
  Automatable : yes
  @critical

🔴 A subscription outside the grace period (current_period_end more than 3 days ago) blocks access
  Scenario    : Insert a subscription row with status=active but current_period_end = now() - 4 days (outside grace). Make a request to a protected route.
  Pass if     : Request is blocked with 402.
  Fail if     : Request succeeds (200) — expired subscription incorrectly grants access.
  Automatable : yes
  @critical

  Fail if     : India plan is applied based solely on user-supplied request body parameters, bypassing server-side geolocation.

  @critical

🔴 A geolocation API failure does not block access to the pricing page or plan selection
  Scenario    : Inject a failing HTTP client into the geolocation service (simulating ip-api.com timeout/503). Call GET /api/payment/plans.

  Fail if     : Endpoint returns 5xx, or returns an empty plan list, or only one plan is available (would block Indian users).
  Automatable : yes (injectable HTTP client)
  @critical

## Subscription Lifecycle State Machine (T-67, T-65)

🟡 Each supported webhook event type (subscription.authenticated, subscription.activated, subscription.pending, subscription.charged, subscription.halted, subscription.cancelled, subscription.completed, payment.failed, refund.processed) triggers the correct DB state transition
  Scenario    : For each event type, send a well-formed webhook payload with a valid signature. Start from an appropriate prior state for each transition (e.g., activated → charged is valid; completed → authenticated is not).
  Pass if     : After each event, the subscriptions row reflects the documented target state (status, current_period_end) for that event type.
  Fail if     : Any event leaves the DB in the wrong state, or throws an unhandled error.
  Automatable : yes
  @functional

  Scenario    : Call POST /api/payment/create-subscription successfully. Immediately query the subscriptions table.
  Pass if     : No row exists in subscriptions for the returned subscriptionId.
  Fail if     : A row was eagerly written to the DB by the create-subscription endpoint.
  Automatable : yes
  @functional

🟡 GET /api/payment/subscription-status returns {active: true} when PAYMENT_ENABLED=false regardless of DB state
  Scenario    : Start server with no RAZORPAY_KEY_ID. Query GET /api/payment/subscription-status.
  Pass if     : Response is {"active": true} and HTTP 200, regardless of what exists in the subscriptions table.
  Fail if     : Response reflects a real DB lookup, returns 402, or errors.
  Automatable : yes
  @functional

🟡 GET /api/payment/plans returns 503 with {"error":"payment_not_configured"} when PAYMENT_ENABLED=false
  Scenario    : Start server with no RAZORPAY_KEY_ID. Call GET /api/payment/plans.
  Pass if     : Response is HTTP 503 with body {"error":"payment_not_configured"}.
  Fail if     : Any other status code or body is returned.
  Automatable : yes
  @functional

🟡 A pre-first-charge subscription (current_period_end IS NULL) is treated as active by the access gate
  Scenario    : Insert a subscription row with status=authenticated and current_period_end=NULL (Razorpay-authenticated but not yet charged). Make a request to a protected route.
  Pass if     : Request succeeds (200) — pre-first-charge access is granted.
  Fail if     : Request is blocked with 402.
  Automatable : yes
  @functional

🟡 verifyWebhookSignature returns false (not throws) on an incorrect signature, and the server never exposes stack traces to the caller
  Scenario    : POST /api/payment/webhook with a structurally valid payload and an incorrect signature.
  Pass if     : HTTP 400 or 401 with a generic error message body. No stack trace or internal detail in the response.
  Fail if     : HTTP 500 (signature error bubbles as uncaught exception), or stack trace is included in the response body.
  Automatable : yes
  @functional

🟡 Reconciliation job corrects a subscription whose DB state drifts from Razorpay's authoritative state
  Scenario    : Insert a subscriptions row with status=active. Mock Razorpay fetchSubscription to return status=halted for the same subscription ID. Trigger the reconciliation job manually.
  Pass if     : After the job completes, the DB row shows status=halted and a log entry records the drift correction.

  @functional

  Scenario    : Insert a subscriptions row with a razorpay_subscription_id that returns 404 from Razorpay (mocked). Run the reconciliation job.
  Pass if     : The orphaned row is deleted from the DB after the job run.
  Fail if     : Orphaned row persists, or the job throws and leaves partial state.
  Automatable : yes
  @functional
