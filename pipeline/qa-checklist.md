## LiveView — WebSocket Connection and Tick Display

🔴 Connecting to /ws/ticks shows a "Connected" status pill; disconnecting shows a "Disconnected" or "Reconnecting" state
  Scenario    : Mount LiveView with a running backend. Observe the status pill immediately after mount, then kill the WebSocket server-side and observe the status change.
  Pass if     : Pill reads "Connected" (or equivalent) within a few seconds of mount; switches to a disconnected/reconnecting label when the connection drops.
  Fail if     : Pill stays in a loading/connecting state indefinitely when the server is up, or stays "Connected" after the server closes the connection.
  Automatable : yes
  @critical

🔴 The LTP display is NOT labeled as real straddle data — the UI makes clear the value is synthetic/simulated
  Scenario    : Mount LiveView while the server sends synthetic random-walk tick messages. Read every label, tooltip, or heading near the LTP value.
  Pass if     : No label uses the words "straddle value", "real price", "live straddle", or any phrasing implying real market data. A "(simulated)" or "index tick" qualifier, or the absence of any straddle label, is acceptable.
  Fail if     : The UI presents the synthetic LTP value under a label that a trader could reasonably read as real option straddle data.
  Automatable : yes
  @critical

🔴 The straddle/latest stub returns null — the UI renders a graceful "feed not yet connected" notice and does not show a misleading value
  Scenario    : Mount LiveView while GET /api/straddle/latest returns {data: null, message: "..."}. Observe what is rendered in the straddle section.
  Pass if     : A human-readable notice (e.g. "Feed not yet connected", "Straddle data unavailable") is displayed. No numeric value is shown. No uncaught error.
  Fail if     : A zero, NaN, undefined, or blank numeric field is shown where straddle data would appear, or an unhandled runtime error is thrown.
  Automatable : yes
  @critical

🔴 Incoming tick messages are parsed with the correct running shape — {type, symbol, ltp, timestamp} — and ltp is rendered as a number
  Scenario    : Send a well-formed tick message: {type: "tick", symbol: "NSE:NIFTY50-INDEX", ltp: 22456.75, timestamp: <ISO string>}. Observe the LTP display.
  Pass if     : The displayed value matches the ltp field exactly (or formatted to a fixed decimal). No NaN, no "[object Object]", no blank.
  Fail if     : The component crashes, shows NaN, or renders a raw JSON string in the value field.
  Automatable : yes
  @critical

🔴 WebSocket reconnects automatically after a dropped connection without creating a storm of simultaneous connections
  Scenario    : Mount LiveView. Drop the WebSocket from the server side. Wait 5 seconds. Observe network panel and connection count.
  Pass if     : Exactly one reconnect attempt fires after ~3 seconds. Only one WebSocket connection is ever open at a time.
  Fail if     : Multiple simultaneous connections are opened, or no reconnect is attempted, or reconnect attempts fire faster than the configured delay.
  Automatable : partial (network panel observation is manual; single-socket assertion can be unit-tested)
  @critical

🔴 Unmounting LiveView closes the WebSocket — no active connection or interval remains after the component is gone
  Scenario    : Mount LiveView, confirm it is connected. Switch to the Trades tab (unmount LiveView). Wait 10 seconds. Observe network connections and console.
  Pass if     : The WebSocket connection closes on unmount. No error messages appear in the console. No setInterval or setTimeout callback fires after unmount.
  Fail if     : The WebSocket stays open after unmount, or a "Can't perform a React state update on an unmounted component" warning appears.
  Automatable : partial (WebSocket close can be unit-tested with a mock; React StrictMode double-mount can be exercised in Vitest)
  @critical

🟡 React StrictMode double-mount does not open two concurrent WebSocket connections
  Scenario    : Run the app in development mode (StrictMode active). Mount LiveView. Inspect network connections.
  Pass if     : Only one WebSocket connection is ever open at a time. The second mount attempt during StrictMode's effect cycle does not leave a dangling socket.
  Fail if     : Two simultaneous WebSocket connections are observed, or a reconnect storm starts because the cleanup from the first mount effect did not cancel the second.
  Automatable : yes (unit test with mock WebSocket)
  @functional

🟡 The chart tick buffer is bounded — adding more ticks than the configured maximum does not grow the buffer indefinitely
  Scenario    : Send 500 tick messages into the component (via unit test or a test harness). Read the buffer size after each insertion.
  Pass if     : The buffer length never exceeds the configured maximum (e.g., 100). Oldest ticks are evicted when the cap is reached.
  Fail if     : Buffer length equals the total number of ticks received (unbounded growth), or the chart update causes a visible memory warning in DevTools.
  Automatable : yes
  @functional

🟡 The last-update timestamp shown in the UI is correct IST and updates with each new tick
  Scenario    : Send a tick with a known UTC timestamp (e.g., 2026-05-23T04:30:00Z, which is 10:00 IST). Observe the displayed timestamp.
  Pass if     : The displayed time reads 10:00 AM IST (or equivalent locale-aware IST string). Sending a second tick updates the displayed time.
  Fail if     : Time is shown in UTC, in a wrong zone, or does not update after each new tick.
  Automatable : yes
  @functional

🟡 The Lightweight Charts line chart renders correctly with zero ticks, one tick, and many ticks
  Scenario    : Render LiveView with (a) an empty buffer, (b) exactly one tick, (c) 50 ticks. Observe the chart container.
  Pass if     : (a) chart renders without error (may be blank or show placeholder), (b) a single point renders without a crash, (c) a line of 50 points renders correctly.
  Fail if     : Any of the three states causes a runtime error or an invisible/blank chart when data is present.
  Automatable : partial (unit: yes; visual regression: no)
  @functional

🟢 The connection-status pill is visually distinct — green for connected, red/amber for disconnected
  Scenario    : Toggle the connection state. Compare pill colors.
  Pass if     : Color change is clearly perceptible. No WCAG blocker (text on background has sufficient contrast).
  Automatable : no
  @non-blocker

🟢 The chart clears or resets correctly when switching away from LiveView and back
  Scenario    : Accumulate 30 ticks in LiveView, switch to Trades tab, switch back to LiveView.
  Pass if     : Chart either retains the existing buffer (expected) or clears cleanly — either is acceptable, but no crash or visual corruption occurs.
  Automatable : partial
  @non-blocker

## TradesView — Trade Table Rendering and Polling

🔴 NUMERIC string fields (gross_pnl, net_pnl, straddle_at_entry) are parsed to numbers before display — no raw string rendered in P&L columns
  Scenario    : Mock GET /api/trades to return a PaperTrade with gross_pnl: "1234.50", net_pnl: "-45.00", straddle_at_entry: "22456.75". Render TradesView and read the P&L cells.
  Pass if     : Cells show numeric values (e.g., 1,234.50 / -45.00 / 22,456.75). No trailing quotes, no raw string that appears unformatted.
  Fail if     : Cells show "1234.50" as a plain unformatted string, or the parseFloat call is absent and JavaScript coerces the string unexpectedly (e.g., "1234.50" + 1 = "1234.501").
  Automatable : yes
  @critical

🔴 Negative P&L is colored red and positive P&L is colored green — color is driven by the numeric sign, not the string content
  Scenario    : Render one trade with net_pnl: "-100.00" and another with net_pnl: "250.00". Observe the CSS class or color applied.
  Pass if     : Negative value has a red class applied; positive value has a green class applied.
  Fail if     : Color is missing, or both rows share the same color, or a string comparison (e.g., net_pnl.startsWith("-")) is used in a way that breaks for edge values like "-0.00" or "0.00".
  Automatable : yes
  @critical

🔴 Trades with null or missing P&L (open trades) do not crash or show NaN — they render a meaningful placeholder
  Scenario    : Mock GET /api/trades to return one trade where gross_pnl and net_pnl are null (open position). Render TradesView.
  Pass if     : P&L cells show a placeholder such as "—", "open", or blank. No NaN, no crash, no "undefined" text.
  Fail if     : Cell shows NaN or undefined, or an uncaught TypeError is thrown during parseFloat(null).
  Automatable : yes
  @critical

🔴 When GET /api/trades returns an empty array, TradesView shows a clear "No trades yet" empty state — not a broken or blank table
  Scenario    : Mock GET /api/trades to return {data: [], message: "no trades"}. Render TradesView.
  Pass if     : A human-readable empty-state message is displayed. The table header (if present) renders without errors. No crash.
  Fail if     : A table with zero rows and no empty-state text renders, leaving the user unable to distinguish "no data" from a loading failure.
  Automatable : yes
  @critical

🔴 When GET /api/trades returns a 500 error, TradesView shows an error notice — it does not crash or silently show stale data as current
  Scenario    : Mock GET /api/trades to respond with HTTP 500. Render TradesView and wait for the poll cycle.
  Pass if     : An error message is rendered (e.g., "Failed to load trades"). No uncaught exception. If previous data was cached, it is either clearly labeled as stale or cleared.
  Fail if     : The component crashes, silently shows an empty table, or shows stale data without any indication that the latest fetch failed.
  Automatable : yes
  @critical

🟡 Entry times are displayed in IST, not UTC
  Scenario    : Mock a trade with entry_time: "2026-05-23T04:00:00.000Z" (9:30 AM IST). Render TradesView and read the entry-time cell.
  Pass if     : The displayed time reads 9:30 AM IST (or equivalent IST representation). Not 04:00 UTC.
  Fail if     : UTC time is shown, or the time is off by exactly 5.5 hours (IST offset is +05:30, a common rounding mistake).
  Automatable : yes
  @functional

🟡 Status badges render for all documented status values — at minimum "open", "closed", and any exit-reason variants
  Scenario    : Mock trades with status values: "open", "closed". Render TradesView. Verify each renders a visible badge.
  Pass if     : Each status value produces a labeled badge. No status value causes a blank cell or a "undefined" string.
  Fail if     : An unknown or unrecognised status value silently renders blank, or only one badge variant exists for all states.
  Automatable : yes
  @functional

🟡 The polling interval does not fire overlapping requests — a slow response from /api/trades is not overridden by a second request before the first completes
  Scenario    : Delay the mock server response for /api/trades by 12 seconds (longer than the 10-second poll). Observe whether a second request fires before the first resolves.
  Pass if     : The second poll fires only after the first has settled (succeeded or failed). No two in-flight requests overlap.
  Fail if     : A second fetch is initiated while the first is still pending, leading to a race condition.
  Automatable : yes (unit-testable with fake timers and a mock fetch)
  @functional

🟡 Unmounting TradesView cancels the in-flight fetch and the polling interval — no state update occurs after unmount
  Scenario    : Mount TradesView. After the first poll fires (but before it resolves), switch tabs (unmount). Observe console for React warnings.
  Pass if     : AbortController cancels the in-flight fetch. No "state update on unmounted component" warning in console.
  Fail if     : A React unmount warning is logged, or a network request continues and resolves after the component is gone.
  Automatable : yes
  @functional

🟡 When GET /api/trades returns a 404, TradesView shows an error notice rather than an empty table
  Scenario    : Mock GET /api/trades to respond with HTTP 404. Render TradesView.
  Pass if     : An explicit error message is displayed. The 404 is not silently treated as "no trades".
  Fail if     : 404 is swallowed and the component renders an empty-state notice as if the endpoint simply returned zero rows.
  Automatable : yes
  @functional

🟢 Exit reason column is populated for closed trades and blank or dashed for open trades
  Scenario    : Render a mix of open and closed trades with exit_reason populated on closed rows and null on open rows.
  Pass if     : Closed-trade rows show the exit reason; open-trade rows show "—" or blank.
  Automatable : yes
  @non-blocker

🟢 Straddle-at-entry column shows a formatted number, not scientific notation, for large values
  Scenario    : Render a trade with straddle_at_entry: "22456.75".
  Pass if     : Displayed as "22,456.75" or "22456.75" — not "2.245675e+4".
  Automatable : yes
  @non-blocker

## PnlView — Client-Side P&L Computation and Chart

🔴 Total net P&L is computed by summing numeric-parsed net_pnl values — string concatenation does not occur
  Scenario    : Mock /api/trades to return three closed trades with net_pnl: "100.00", "200.50", "-50.25". Render PnlView and read the "Total Net P&L" figure.
  Pass if     : Displayed total is 250.25 (correct numeric sum). Not "100.00200.50-50.25" (string concatenation).
  Fail if     : Total is wrong due to implicit string coercion, or is NaN due to a missing parseFloat call on the string fields.
  Automatable : yes
  @critical

🔴 Open trades (null net_pnl) are excluded from the total P&L sum — they do not contribute NaN or zero to the aggregate
  Scenario    : Mock /api/trades to return one closed trade (net_pnl: "300.00") and one open trade (net_pnl: null). Render PnlView.
  Pass if     : Total net P&L shows 300.00. Open trade contributes nothing to the sum.
  Fail if     : Total is NaN (because null was passed to parseFloat without a guard), or total is incorrectly 300.00 + 0 = 300.00 for the wrong reason (masked null → 0 instead of skip).
  Automatable : yes
  @critical

🔴 "Today's P&L" uses IST date boundaries — a trade that closed at 23:59 UTC on May 22 (04:29 IST May 23) is counted in the correct IST day
  Scenario    : Mock a trade with exit_time: "2026-05-22T18:29:00.000Z" (23:59 IST May 22) and another with exit_time: "2026-05-22T18:31:00.000Z" (00:01 IST May 23). Set system date to May 23 IST. Render PnlView.
  Pass if     : The 23:59 IST trade is counted in May 22's P&L (yesterday) and the 00:01 IST trade is counted in today's P&L.
  Fail if     : Both trades fall in the same UTC day and are both counted as "today", or the IST offset (+05:30) is applied incorrectly.
  Automatable : yes (unit test with mocked Date.now)
  @critical

🔴 When GET /api/trades returns an error or empty data, PnlView shows a clear empty/error state — not a zero P&L dashboard that implies valid data
  Scenario    : Mock /api/trades to return HTTP 500. Render PnlView.
  Pass if     : An error message is displayed. No P&L figures are shown (or all figures are clearly marked unavailable). No crash.
  Fail if     : PnlView renders P&L as "0.00" and win rate as "0%" as if the data genuinely showed no activity — this is misleading to a trader.
  Automatable : yes
  @critical

🟡 Win rate is computed as closed-wins divided by total-closed (not total-all) — open trades are excluded from the denominator
  Scenario    : Mock 3 closed trades (2 profitable, 1 loss) and 2 open trades. Render PnlView and read the win rate.
  Pass if     : Win rate = 2/3 = 66.7%. Open trades do not inflate the denominator.
  Fail if     : Win rate = 2/5 = 40% (open trades counted) or any other incorrect value.
  Automatable : yes
  @functional

🟡 The cumulative P&L chart uses exit_time for ordering closed trades — not entry_time or insertion order
  Scenario    : Mock 3 closed trades with exit times out of chronological order in the API response. Render PnlView.
  Pass if     : The chart plots points in exit_time order. The final cumulative value equals the arithmetic sum of all closed net_pnl values.
  Fail if     : Points are plotted in response-array order (which may differ), or the cumulative total is wrong.
  Automatable : yes
  @functional

🟡 The cumulative P&L chart renders without error when there are zero closed trades (only open trades, or no trades at all)
  Scenario    : Mock /api/trades to return only open trades (status: "open") or an empty array. Render PnlView.
  Pass if     : Chart renders without crash. An empty-state or "no closed trades" message is shown. No Lightweight Charts error in console.
  Fail if     : The chart library throws when given an empty data set, or an unhandled runtime error occurs.
  Automatable : yes
  @functional

🟡 Open-position count and closed-position count are displayed separately and accurately
  Scenario    : Mock 3 open trades and 5 closed trades. Render PnlView.
  Pass if     : Open count = 3, Closed count = 5. Both figures are labeled clearly.
  Fail if     : A single total count is shown, or the open/closed split does not match the mock data.
  Automatable : yes
  @functional

🟡 PnlView unmount cancels its fetch and polling — no state update fires after the component is gone
  Scenario    : Mount PnlView. While the first fetch is in-flight, switch to a different tab. Observe console.
  Pass if     : AbortController cancels the fetch. No "state update on unmounted component" warning.
  Fail if     : A React warning is logged, or a fetch response is processed and setState is called on the unmounted component.
  Automatable : yes
  @functional

🟢 Total net P&L is colored red for negative, green for positive, neutral for zero
  Scenario    : Render PnlView with total net P&L of 500.00, then -200.00, then 0.00.
  Pass if     : Green for positive, red for negative, neutral (gray or white) for zero.
  Automatable : yes
  @non-blocker

🟢 The chart displays exit_time in IST on its x-axis labels, not UTC
  Scenario    : Render PnlView with closed trades. Read the x-axis tick labels on the cumulative P&L chart.
  Pass if     : Labels reflect IST times (or IST dates). Not raw UTC strings.
  Automatable : no (visual)
  @non-blocker
