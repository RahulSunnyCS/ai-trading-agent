## T-54 — Fyers Historical REST Client

🔴 Missing Fyers credentials produce a loud startup error, not a silent no-op
  Scenario    : Start the historical client with FYERS_APP_ID and FYERS_ACCESS_TOKEN unset; attempt to initiate any fetch call
  Pass if     : A clearly labelled error is thrown/logged before any HTTP request is attempted; the process does not silently continue
  Fail if     : The client proceeds to make an unauthenticated request, returns empty data without error, or swallows the missing-credential condition
  Automatable : yes
  @critical

🔴 A 401 response mid-backfill writes a checkpoint and emits a fail-loud resumable marker
  Scenario    : Inject a mock Fyers REST endpoint that returns HTTP 401 partway through a multi-chunk date range; observe the client's response
  Pass if     : The `backfill_ranges` row for the interrupted symbol+range is updated with a resumable status and checkpoint timestamp before the client exits; an error is logged with explicit guidance to re-auth
  Fail if     : The client writes a "complete" status despite the 401, continues silently with partial data, swallows the error, or leaves no checkpoint
  Automatable : yes
  @critical

🔴 HTTP 429 triggers exponential backoff and the fetch eventually succeeds
  Scenario    : Mock the Fyers REST endpoint to return 429 twice then 200; run a single-chunk fetch
  Pass if     : The client retries after each 429 with increasing delay; the third call succeeds; the data is written correctly; no error is surfaced to the caller
  Fail if     : The client treats 429 as a fatal error and aborts, or retries with zero delay (tight loop), or succeeds but writes partial/duplicate data
  Automatable : yes
  @critical

🔴 The client only connects to the fixed Fyers host; no user-supplied URL is accepted
  Scenario    : Inspect the client source for any code path that constructs an HTTP URL from caller-supplied input; attempt to call the client with a crafted symbol value containing a URL
  Pass if     : All HTTP calls target a hard-coded Fyers base URL; no symbol or parameter value is interpolated into the host or scheme portion of any request
  Fail if     : Any outbound HTTP call uses a URL derived from user/caller input (SSRF vector)
  Automatable : partial
  @critical

🟡 Date-range chunking respects Fyers per-resolution day caps
  Scenario    : Request a date range that exceeds the per-resolution Fyers cap (e.g. >100 days for 1-minute resolution); observe the number and boundaries of HTTP requests made
  Pass if     : The range is split into chunks each within the Fyers cap; no single request spans more days than the documented limit; chunk boundaries are contiguous with no gaps or overlaps
  Fail if     : A single oversized request is sent, or chunks have gaps/overlaps between them
  Automatable : yes
  @functional

🟡 Response typing covers all fields; unknown fields do not cause a runtime panic
  Scenario    : Mock the Fyers REST endpoint to return an extra unknown field alongside valid OHLCV data
  Pass if     : The client parses known fields correctly and ignores unknown fields without throwing
  Fail if     : The client throws a type error or discards the entire response because of an unexpected field
  Automatable : yes
  @functional

🟢 Token reuse: the client calls loadStoredToken(pool) exactly once per backfill session, not per chunk
  Scenario    : Run a multi-chunk backfill session against a mocked endpoint; spy on loadStoredToken calls
  Pass if     : loadStoredToken is called once at session start and the token is reused across all chunks
  Fail if     : loadStoredToken is called on every chunk (redundant DB reads)
  Automatable : yes
  @non-blocker

## T-55 — Historical Backfill Store + Writer (migration 007)

🔴 The partial unique index on (symbol, time) WHERE source='fyers-historical' prevents duplicate rows on re-run
  Scenario    : Run a backfill for a symbol+range; run the identical backfill a second time without clearing the table
  Pass if     : The second run writes zero new rows; the row count in market_ticks for that symbol+range remains identical after both runs; no constraint violation error is thrown (idempotent upsert)
  Fail if     : The second run inserts duplicate rows, or throws an unhandled unique-violation error that aborts the backfill
  Automatable : yes
  @critical

🔴 The partial unique index is disjoint from live-ingestion keys (source='fyers' / 'simulator' are unaffected)
  Scenario    : Insert a live-ingestion row with source='fyers' and the same (symbol, time) as a previously backfilled row with source='fyers-historical'
  Pass if     : Both rows coexist; no constraint violation; neither insert is rejected
  Fail if     : The partial unique index incorrectly blocks the live-ingestion row, or the live-ingestion row overwrites/conflicts with the backfilled row
  Automatable : yes
  @critical

🔴 Backfill write rejected (fail-loud) if a required CE or PE option leg candle is absent
  Scenario    : Provide index candles for a symbol+range but omit the CE leg candle for one step; trigger the straddle reconstruction write path
  Pass if     : The writer throws an explicit error naming the missing leg and the timestamp; no zero-fill or interpolated row is written; the backfill_ranges entry is marked with a gap, not "complete"
  Fail if     : A zero or interpolated value is silently written; the step is skipped without marking a gap; no error is surfaced
  Automatable : yes
  @critical

🔴 An INVALID index created during migration 007 is detected and cleaned up on failure
  Scenario    : Simulate a migration 007 run that is interrupted after the index is created but before it is validated; restart the migration
  Pass if     : The runner detects the INVALID index state on restart, drops it, and re-creates it cleanly; the migration completes successfully
  Fail if     : The INVALID index is left in place; subsequent queries silently ignore it or throw an error; the migration appears to succeed with a broken index
  Automatable : partial
  @critical

🔴 All backfill writes include a bounded time filter (hypertable discipline — no unbounded INSERT)
  Scenario    : Inspect or instrument every INSERT path in the backfill writer; verify each carries an explicit time column value in the bounded range being processed
  Pass if     : Every insert carries an explicit time-range-bounded value; no INSERT omits the time column
  Fail if     : Any INSERT path omits the time column or inserts rows without a time filter, risking a full-hypertable scan on subsequent queries
  Automatable : partial
  @critical

🔴 Resume-after-interruption continues from the checkpoint, not from the beginning
  Scenario    : Begin a large backfill, interrupt it at the midpoint (kill the process), then restart the backfill for the same symbol+range
  Pass if     : The restart reads the checkpoint from backfill_ranges and begins from the last completed chunk; no already-written chunks are re-fetched; total rows written matches a fresh run
  Fail if     : The restart begins from the beginning of the range, re-fetching and attempting to insert already-written chunks (duplicate-insert attempt or wasted API calls)
  Automatable : yes
  @critical

🟡 Trading-calendar gaps (NSE holidays, expiry-morning half-days) are marked in backfill_ranges, not silently "complete"
  Scenario    : Run a backfill over a date range that includes a known NSE holiday (e.g. Diwali Muhurat trading day) and an F&O expiry morning; inspect the resulting backfill_ranges rows
  Pass if     : Each calendar gap appears as a distinct backfill_ranges row with an explicit gap status; no gap is merged into an adjacent "complete" chunk
  Fail if     : The calendar gap is silently treated as complete, marked with rows_written=0 and status="complete", or not recorded at all
  Automatable : yes
  @functional

🟡 The resolution column is present and correctly tagged on every backfilled row
  Scenario    : Backfill the same symbol at two different resolutions (e.g. 1-minute and daily); query market_ticks and filter by each resolution value
  Pass if     : Every row carries the correct resolution tag; no row has a null or incorrect resolution; the two sets are disjoint by resolution value
  Fail if     : Any row has a null resolution, or rows from different resolutions cannot be distinguished by the column value
  Automatable : yes
  @functional

🟡 The backfill_ranges table correctly tracks rows_written and updated_at across chunks
  Scenario    : Run a multi-chunk backfill; after completion, query backfill_ranges for the symbol+range
  Pass if     : rows_written matches the actual row count in market_ticks for that source/range; updated_at is more recent than the start time; status is "complete"
  Fail if     : rows_written is zero or incorrect; updated_at is not updated; status disagrees with the actual data state
  Automatable : yes
  @functional

🟢 Migration 007 is idempotent — running it twice against an already-migrated database does not error
  Scenario    : Apply migration 007 to a fresh database; apply it again
  Pass if     : The second application is a no-op; schema_migrations records a single entry; no error is thrown
  Automatable : yes
  @non-blocker

## T-56 — Historical Straddle Reconstruction

🔴 ATM strike selection at each reconstruction step uses only data at-or-before that step's timestamp (no look-ahead)
  Scenario    : Construct a synthetic candle sequence where the index price changes sharply at step T+1; run reconstruction; inspect the ATM strike chosen at step T
  Pass if     : The ATM strike at step T reflects only the index candle at-or-before T; the strike does not reflect the T+1 price
  Fail if     : The ATM strike at any step T is derived from a candle with timestamp > T; any future bar is consumed before its clock time
  Automatable : yes
  @critical

🔴 Reconstruction uses the same extracted pure compute function as the live StraddleCalculator (single implementation, not a fork)
  Scenario    : Inspect the import chain of the reconstruction module; verify computeRoc and computeAcceleration are imported from straddle-calc.ts, not re-implemented inline
  Pass if     : The reconstruction module imports and calls the identical computeRoc/computeAcceleration exports from src/ingestion/straddle-calc.ts; no duplicate implementation exists
  Fail if     : The reconstruction module contains its own ROC or acceleration calculation, or the computation diverges from the live path in any way (even by rounding or type coercion)
  Automatable : partial
  @critical

🔴 Missing CE or PE leg candle causes a fail-loud error, never a zero-fill or interpolation
  Scenario    : Run reconstruction over a range where one step has no CE candle in option_ticks; observe the reconstruction output for that step
  Pass if     : An explicit error is thrown identifying the missing leg, strike, and timestamp; no straddle_snapshots row is written for that step; the step is recorded as a gap
  Fail if     : A straddle_snapshots row is written with zero, interpolated, or previous-close CE/PE price; the gap is silently skipped
  Automatable : yes
  @critical

🟡 Reconstructed straddle_snapshots rows are tagged with the source resolution used
  Scenario    : Run reconstruction using 1-minute resolution candles; query the resulting straddle_snapshots rows
  Pass if     : Every written row carries the correct source resolution tag; rows from different resolution runs are distinguishable
  Fail if     : Any row has a null or incorrect resolution tag
  Automatable : yes
  @functional

🟡 Reconstruction output for a known synthetic input matches hand-computed straddle values
  Scenario    : Provide a fully deterministic synthetic candle sequence with known CE+PE values; run reconstruction; compare output straddle_values, ROC, and acceleration against hand-computed expected values
  Pass if     : Every reconstructed straddle_value equals cePrice + pePrice for that step; ROC and acceleration match hand-computed values to the precision used in computeRoc/computeAcceleration
  Fail if     : Any field differs from the expected value beyond floating-point rounding tolerance
  Automatable : yes
  @functional

🟢 Reconstruction skips snapshot steps where the underlying index candle is absent, and logs a warning
  Scenario    : Run reconstruction over a range with one missing underlying index candle (e.g. a thin trading period)
  Pass if     : The step is skipped with a debug/warning log; no error is thrown; reconstruction continues for subsequent steps
  Automatable : yes
  @non-blocker

## T-57 — Deterministic Replay Harness

🔴 The golden-replay oracle: the frozen fixture replays to a structurally identical trade ledger 100 times in succession
  Scenario    : Load the checked-in frozen historical fixture; run the replay harness 100 times consecutively against it; collect the resulting trade ledger after each run; compare all 100 ledgers
  Pass if     : All 100 ledgers are structurally identical (canonicalized JSON with decimal.js-normalized numeric fields and stable key ordering); no ledger differs in any typed field
  Fail if     : Any single run produces a ledger that differs from the first run in trade count, exit reason, entry/exit timestamps, or any numeric field beyond declared tolerance; or any run deadlocks/times out
  Automatable : yes
  @critical

🔴 The replay code path contains zero floating (un-awaited) promises
  Scenario    : Audit all async calls in HistoricalFeed, the replay driver, and any directly-called helpers; inspect for bare void or fire-and-forget async calls; run the replay under a test that catches unhandled-rejection events
  Pass if     : No bare void or un-awaited async call exists in the replay code path; the unhandled-rejection handler is never triggered during a replay run
  Fail if     : Any async call in the replay path is not awaited; an unhandled promise rejection occurs during replay; the replay produces non-deterministic output traceable to a race condition
  Automatable : partial
  @critical

🔴 No '$' Redis cursor is used in the replay path; consumers start from a fixed ID ('0')
  Scenario    : Inspect the HistoricalFeed and any replay-path Redis consumers for the cursor value passed to XREAD or XREADGROUP; run the replay and assert the cursor value used at start
  Pass if     : Every Redis consumer in the replay path starts with cursor '0' or a specific numeric ID; no consumer uses '$' as a starting cursor
  Fail if     : Any replay-path consumer passes '$' to XREAD or XREADGROUP, which would cause it to miss ticks published before it started reading
  Automatable : partial
  @critical

🔴 The named drain-barrier (PositionMonitor.processedThrough or XPENDING==0 assertion) blocks clock.advance() until all events are consumed
  Scenario    : Inject a synthetic tick sequence; run the replay driver; verify that clock.advance() is never called before the drain-barrier resolves for the current step; instrument the barrier to record call order
  Pass if     : clock.advance() is always preceded by a resolved drain-barrier for the current step; no tick from step N+1 is processed before all step-N events are fully consumed
  Fail if     : clock.advance() is called before the drain-barrier resolves; any step-N+1 tick is processed concurrently with step-N events; the trade ledger shows out-of-order exits
  Automatable : yes
  @critical

🔴 StraddleCalculator.snapshotStep() is awaited by the replay driver and resolves only after the snapshot is written to straddle.values
  Scenario    : Instrument snapshotStep() to record the Redis XADD call; run the replay driver for multiple steps; verify the driver's await completes only after the XADD
  Pass if     : The replay driver's await on snapshotStep() always unblocks after the corresponding XADD to straddle.values; no clock advance occurs before the snapshot is in the stream
  Fail if     : The driver proceeds to the next step before the snapshot write is confirmed; any step's snapshot is missing from straddle.values when the drain-barrier is checked
  Automatable : yes
  @critical

🔴 Bidirectional SIMULATE-path vs replay-path ledger equivalence: replaying SIMULATE's own recorded ticks reproduces an identical ledger
  Scenario    : Run the SIMULATE pipeline for a short fixed period; capture all ticks emitted to market.ticks; replay those exact ticks through the HistoricalFeed; compare the two resulting trade ledgers
  Pass if     : The replay-path ledger is structurally identical to the SIMULATE-path ledger for the same tick sequence (same trades, same entry/exit values, same exit reasons)
  Fail if     : The two ledgers differ in any structural field; any trade present in one path is absent or different in the other
  Automatable : yes
  @critical

🔴 The replay fixture includes at least one gap-marked range and at least one resolution tag, making it a valid M3b input
  Scenario    : Inspect the checked-in golden fixture file; verify its schema includes a backfill_ranges entry with gap status and at least one straddle_snapshots row with a non-null resolution tag
  Pass if     : Both conditions are present in the fixture file
  Fail if     : The fixture contains no gap-marked range or no resolution tag, making it an invalid M3b entry criterion
  Automatable : yes
  @critical

🟡 Existing straddle-calc.test.ts passes unchanged after the shared-compute-function extraction refactor
  Scenario    : Run the full straddle-calc.test.ts suite after the T-57 refactor that extracts computeRoc/computeAcceleration into the shared path; compare pass/fail counts before and after
  Pass if     : All tests that passed before the refactor still pass; no test is silently skipped or modified; the suite reports identical counts

  Scenario    : Run the full position-monitor.test.ts suite after the T-57 refactor; compare pass/fail counts before and after
  Pass if     : All tests that passed before the refactor still pass; no test is silently skipped or modified
  Fail if     : Any previously-passing position-monitor test fails or is modified to accommodate the refactor

🟡 Live SIMULATE snapshot cadence is unaffected by the refactor (15-second interval still fires on wall-clock schedule)

## T-51 — Backtest Runner [M3b — GATED BEHIND M2: requires T-27 personality router]

🔴 [M3b] The backtest HTTP endpoint enqueues a BullMQ job and returns a job ID immediately; it does not run inline
  Scenario    : POST to the backtest trigger endpoint with a valid date range; measure response time and check the database for immediate changes
  Pass if     : Response arrives within 200ms; response body contains a job_id; no backtest_results rows exist yet; the BullMQ queue contains exactly one pending job
  Fail if     : Response takes longer than the backtest duration (inline execution); response contains completed results; the job is not enqueued; any synchronous DB write to backtest_results occurs before the response

  @critical

🔴 [M3b] The date-range cap is enforced; requests exceeding the maximum are rejected before enqueueing
  Scenario    : POST to the backtest endpoint with a date range that exceeds the configured hard cap; observe the response
  Pass if     : HTTP 422 (or equivalent rejection) is returned with an explicit error describing the cap; no job is enqueued; no rows are written
  Fail if     : The oversized job is enqueued; or the request is accepted and the cap is only enforced inside the worker (resource exhaustion window); or no error is returned to the caller
  Automatable : yes
  @critical

🔴 [M3b] Every hypertable read in the backtest runner carries a bounded time filter; unbounded reads are rejected
  Scenario    : Instrument or inspect every SQL query issued by the backtest runner; attempt to construct a code path that queries market_ticks or straddle_snapshots without a WHERE time filter
  Pass if     : No SQL query against a hypertable is issued without a bounded time filter; any attempt to issue an unbounded query throws an explicit error before execution
  Fail if     : Any hypertable query in the backtest runner executes without a time-range filter, which would scan the full hypertable
  Automatable : partial
  @critical

🔴 [M3b] The HOLDOUT period is never included in any training or tuning query

  Pass if     : No query to backtest_results, market_ticks, or straddle_snapshots uses a date within the holdout range for training or tuning purposes; the holdout boundary is a hard constraint, not a soft hint
  Fail if     : Any training or tuning consumer receives data from within the holdout period; the holdout boundary can be overridden by a caller-supplied parameter
  Automatable : yes
  @critical

🟡 [M3b] The backtest runner iterates all ACTIVE personalities, not a hard-coded list
  Scenario    : Insert a second active personality into personality_configs; trigger a backtest run; inspect backtest_results
  Pass if     : backtest_results contains rows for each active personality; the Clockwork personality is included; no personality is omitted
  Fail if     : Only a hard-coded personality is processed; a newly added personality produces no rows; the Clockwork personality is skipped
  Automatable : yes
  @functional

🟡 [M3b] Single BullMQ concurrency prevents simultaneous backtest runs
  Scenario    : Enqueue two backtest jobs in rapid succession; observe job execution
  Pass if     : The second job remains in the queue until the first completes; only one backtest worker executes at a time
  Fail if     : Both jobs execute concurrently, risking resource exhaustion or conflicting writes to backtest_runs

  Pass if     : The status endpoint returns a valid status (queued/running/complete/failed) at each poll; the status transitions correctly; the completed job includes a summary of results
  Fail if     : The status endpoint returns 404 for a valid job_id; status never transitions from queued; the endpoint blocks until completion (not a poll pattern)
  Automatable : yes
  @functional

🟢 [M3b] The backtest_runs table records the date range, personality set, and job metadata for each run
  Scenario    : Complete a backtest run; query backtest_runs for the resulting row
  Pass if     : The row contains the requested date range, the list of personality IDs processed, job start/end timestamps, and final status
  Automatable : yes

  Automatable : yes
  @critical

🟡 The source column is correctly populated ('fyers-historical') for all backfill-written rows and never overwritten by live ingestion paths
