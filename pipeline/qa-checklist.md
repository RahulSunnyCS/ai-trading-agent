## Bayesian Optimizer Min-Sample Gate

🔴 The optimizer returns no suggestion (not a low-confidence suggestion) when post-filter regime-tagged row count is below 200
  Scenario    : Provide the optimizer with a retrospection dataset filtered to poc-consistent rows for one personality, containing 199 regime-tagged rows. Invoke the optimizer.
  Pass if     : Optimizer returns a result indicating "no suggestion / insufficient samples" without generating any candidate parameter value. No mutation is queued to the approval gate.
  Fail if     : Optimizer generates a suggestion (even with a low-confidence flag), or throws an unhandled error rather than returning a clean no-suggestion signal.
  Automatable : yes
  @critical

🔴 The optimizer's minimum-sample threshold references the same shared constant as the rule engine, not a locally re-declared value
  Scenario    : Inspect the source of the minimum-sample constant used in the optimizer code path and in the rule engine code path. They must reference the same exported constant, not separate numeric literals.
  Pass if     : A single exported constant (e.g. MIN_OPTIMIZER_SAMPLES) is imported in both the rule engine and the Bayesian optimizer; no duplicate numeric declaration of 200 exists.
  Fail if     : The optimizer has its own hard-coded 200, or defines a local constant with the same value independently from the rule engine constant.
  Automatable : yes
  @critical

🔴 The optimizer counts only POST-FILTER regime-tagged rows (after freshness + poc-consistency filtering) toward the 200-row minimum, not the raw retrospection row count
  Scenario    : Construct a dataset of 250 raw retrospection rows where 100 rows are poc-inconsistent and 60 are filtered by the freshness check, leaving 90 valid rows. Invoke the optimizer.
  Pass if     : Optimizer evaluates 90 rows against the 200-row threshold and returns no suggestion.
  Fail if     : Optimizer counts 250 (raw), 190 (after one filter), or any count other than 90 (post-all-filters) against the threshold, and generates a suggestion.
  Automatable : yes
  @critical

🔴 The optimizer falls back to the rule engine when GP fails numerically, rather than propagating the exception or emitting a suggestion
  Scenario    : Force a GP Cholesky failure (inject a near-singular covariance matrix). Observe the optimizer's response.
  Pass if     : System silently falls back to the rule engine for that EOD run; a fallback log entry is emitted; no parameter suggestion originates from the failed GP.
  Fail if     : The numerical failure propagates as an uncaught exception, halts the BullMQ job, or causes the optimizer to emit a partially-computed suggestion.
  Automatable : partial
  @critical

  @critical

🔴 The optimizer respects EVOLUTION_REQUIRE_APPROVAL=TRUE: suggestions are queued to the approval gate and not applied directly to personality_configs
