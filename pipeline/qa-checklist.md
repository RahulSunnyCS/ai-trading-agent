## Server Bootstrap and Lifecycle (T-01)

🔴 The server only binds to localhost and refuses to bind to all interfaces
  Scenario    : Start the server in both sim and live mode; inspect the bound address reported by Fastify at startup
  Pass if     : Fastify logs show listening on 127.0.0.1:3000 (or ::1 for IPv6 loopback only)
  Fail if     : Server binds to 0.0.0.0 or any externally reachable interface
  Automatable : yes
  @critical

🔴 GET /health returns 200 with the expected JSON body
  Scenario    : Send GET /health with a valid Origin header; parse the response body
  Pass if     : HTTP 200; body is exactly { "status": "ok" }
  Fail if     : Any non-200 status, or body is missing the status field, or body contains extra sensitive fields
  Automatable : yes
  @critical

🔴 GET /api/status returns mode, underlying, and uptimeMs fields
  Scenario    : Start in sim mode (SIMULATE=true); call GET /api/status
  Pass if     : HTTP 200; body contains { mode: "sim", underlying: string, uptimeMs: number }
  Fail if     : Any field is absent, mode is wrong for the active run mode, or uptimeMs is 0 or negative
  Automatable : yes
  @critical

🔴 Origin-guard blocks requests whose Origin header is not in the allowlist
  Scenario    : Send GET /health with Origin: http://evil.example.com
  Pass if     : HTTP 403 response; request is not processed further
  Fail if     : HTTP 200 or any non-403 successful response
  Automatable : yes
  @critical

🔴 Origin-guard blocks requests whose Host header is not in the allowlist
  Scenario    : Send GET /health with no Origin header but Host: attacker.internal (DNS-rebinding simulation)
  Pass if     : HTTP 403 response
  Fail if     : Request is processed and returns 200
  Automatable : yes
  @critical

🔴 Origin-guard permits requests from all three explicitly allowed origins
  Scenario    : Send requests with each of: Origin: http://localhost:3000, http://localhost:5173, http://127.0.0.1:3000
  Pass if     : All three receive HTTP 200 from /health
  Fail if     : Any allowed origin is incorrectly blocked with 403
  Automatable : yes
  @critical

🟡 GET /api/status returns live mode indicator when not in simulation mode
  Scenario    : Start with SIMULATE unset or false; call GET /api/status
  Pass if     : Body contains { mode: "live" }
  Fail if     : mode is "sim" when running live
  Automatable : partial (requires live credentials or a mock; sim mode is easier to automate)
  @functional

🟡 Graceful shutdown completes the full sequence without hanging
  Scenario    : Start the server; send SIGTERM; observe shutdown log output and process exit code
  Pass if     : Logs show stopTradingLoop → closeAll WS → app.close → closeRedis → closePool in that order; process exits 0 within 7 seconds
  Fail if     : Process hangs beyond 7 seconds, exits non-zero, or skips a shutdown step
  Automatable : partial (subprocess management required)
  @functional

🟢 Server starts without errors in simulation mode with no environment variables except SIMULATE=true
  Scenario    : Set only SIMULATE=true and start the server; observe startup log
  Pass if     : Server reaches "listening" state with no thrown errors
  Fail if     : Crash or unhandled exception at startup
  Automatable : yes
  @non-blocker

## Evolution Guard and Security (T-02)

  Scenario    : Create a pending evolution_approvals row targeting Clockwork's personality_id; POST /api/evolution/:id/approve

  Fail if     : HTTP 200 (approval accepted) or any mutation to Clockwork's row in personality_configs
  Automatable : yes
  @critical

🔴 Approving an evolution for a parameter not in MUTABLE_PARAMETERS returns 400 and makes no database change
  Scenario    : Insert a pending evolution_approvals row where parameter = 'entry_type' (a frozen attribute); POST /api/evolution/:id/approve
  Pass if     : HTTP 400; no rows updated in personality_configs

  @critical

🔴 Concurrent approval of the same pending evolution returns 409 for the second caller

  Fail if     : Both requests succeed (double-apply), or the row is left in an inconsistent state
  Automatable : partial (requires concurrent request tooling)
  @critical

🔴 The auto-apply path in evolution-rules.ts refuses to update a frozen personality parameter
  Scenario    : Call applyEvolutionRules() with a proposal targeting Clockwork (is_frozen=true); observe outcome
  Pass if     : FrozenPersonalityError is thrown; no UPDATE executed on personality_configs

  Automatable : yes
  @critical

🔴 The auto-apply path in evolution-rules.ts refuses to update a parameter outside MUTABLE_PARAMETERS
  Scenario    : Call applyEvolutionRules() with a proposal where parameter = 'management_style'
  Pass if     : Error thrown (parameter not in allowlist); no UPDATE executed
  Fail if     : management_style column is updated
  Automatable : yes
  @critical

🔴 The auto-apply path uses a parameterized column reference, not raw string interpolation

  Fail if     : SQL injection string reaches a database query as an identifier fragment

  @critical

🔴 Comparison-integrity check blocks approval when min_probability drift across Precision/Adjuster/Reducer exceeds 8 percentage points

  @critical

🔴 The approve transaction is atomic — a failure mid-transaction leaves no partial state
  Scenario    : Inject a simulated failure between step 5 (UPDATE personality_configs) and step 6 (UPDATE evolution_approvals); then read both tables
  Pass if     : personality_configs retains its pre-approve value AND evolution_approvals.status remains 'pending'
  Fail if     : personality_configs was updated but evolution_approvals.status still shows 'pending' (split-brain)
  Automatable : partial (requires test-mode fault injection or mock DB)
  @critical

🔴 Migration 003 CHECK constraint on evolution_approvals.parameter prevents insertion of non-allowlist parameter names at the database level
  Scenario    : Attempt direct INSERT into evolution_approvals with parameter = 'management_style'
  Pass if     : PostgreSQL raises constraint violation; row not inserted
  Fail if     : Row inserted successfully
  Automatable : yes
  @critical

🟡 GET /api/evolution/pending returns only rows with status='pending', joined with personality name
  Scenario    : Insert one pending and one applied approval; call GET /api/evolution/pending
  Pass if     : Response array contains exactly one item; it includes personality name
  Fail if     : Applied approval appears in the list, or personality name is absent
  Automatable : yes
  @functional

🟡 POST /api/evolution/:id/approve updates retrospection_results.applied and applied_at as part of the same transaction
  Scenario    : Approve a pending evolution with a valid retrospection_result_id FK; read retrospection_results after
  Pass if     : applied=TRUE and applied_at is a recent timestamp in the same row
  Fail if     : applied remains FALSE or applied_at is NULL
  Automatable : yes
  @functional

🟡 POST /api/evolution/:id/reject flips status to 'rejected' and records the optional reason
  Scenario    : POST /api/evolution/:id/reject with body { reason: "market regime changed" }
  Pass if     : HTTP 200; evolution_approvals.status='rejected'; reason stored

🟡 GET /api/personalities returns all 10 personality rows

## WebSocket Connection Manager and Broadcaster (T-04)

🔴 Trade open and trade close events are never dropped regardless of connection buffer depth
  Scenario    : Connect a single WS client; artificially block its outbound socket to simulate a slow consumer; trigger 30 rapid broadcastSnapshot calls followed by one broadcastTrade('trade_open'); observe which messages the client receives once unblocked
  Pass if     : The trade_open message is delivered; up to 20 snapshot messages may have been dropped; no snapshot delivery is counted as a failure
  Fail if     : The trade_open message is missing from the client's received sequence
  Automatable : partial (requires test harness to block socket writes)
  @critical

🔴 Signal events are never dropped regardless of connection buffer depth
  Scenario    : Same setup as above but trigger broadcastSignal instead of broadcastTrade
  Pass if     : The signal message is delivered; snapshot drops are acceptable
  Fail if     : The signal message is missing
  Automatable : partial
  @critical

🔴 Connecting a 21st client when no existing connection is idle evicts nothing and the new client receives WS close frame 1013
  Scenario    : Open 20 WS connections, all of which have sent a message within the last 60 seconds; attempt a 21st connection
  Pass if     : 21st connection receives WebSocket close code 1013 (Try Again Later); the 20 existing connections remain open
  Fail if     : A non-idle connection is silently evicted, or the 21st client is accepted bringing total above 20
  Automatable : yes
  @critical

🔴 The oldest idle connection (last message > 60s ago) is evicted when a 21st client connects
  Scenario    : Open 20 WS connections; let one go idle for > 60 seconds; open a 21st connection
  Pass if     : The idle connection is closed; the 21st client connects successfully; total active connections remain at 20
  Fail if     : The 21st client is rejected when an idle slot is available, or a non-idle client is evicted
  Automatable : partial (requires time control or mocked timestamps)
  @critical

🔴 A dead socket (no pong within 10s of ping) is removed from the active connection set
  Scenario    : Open a WS connection; stop the client from responding to pings; wait 30s + 10s (one ping interval plus pong timeout)
  Pass if     : The socket is removed from the connection set; subsequent broadcasts do not attempt to write to the dead socket
  Fail if     : Dead socket remains in the set; broadcast throws an unhandled exception on write
  Automatable : partial (requires mock WS client that ignores pings)
  @critical

🔴 broadcastSnapshot is fire-and-forget — it does not await or block the snapshot interval
  Scenario    : Simulate a slow WS write (delayed by 2s); trigger multiple snapshot intervals in rapid succession; measure whether the interval callback completes within 100ms
  Pass if     : Interval callbacks complete without waiting for WS delivery; timestamps show non-blocking behaviour
  Fail if     : Interval callbacks are delayed by the WS write latency (evidence of awaiting)
  Automatable : partial (requires timing instrumentation)
  @critical

🔴 Both live and sim snapshot call sites wire broadcastSnapshot after computeAndSaveSnapshot
  Scenario    : Code inspection / integration test — run in sim mode; connect a WS client; observe that snapshot messages arrive on the client
  Pass if     : WS client receives snapshot messages during a sim run without any code change to sim-specific paths
  Fail if     : Snapshot messages arrive only in live mode, or the sim snapshot call site is not wired
  Automatable : yes (sim mode integration test)
  @critical

🟡 Snapshot messages beyond 20 pending are dropped for a slow consumer, and older snapshots are dropped first
  Scenario    : Connect a throttled WS client; send 25 broadcastSnapshot calls before the client reads; inspect the 20 messages the client eventually receives
  Pass if     : The 20 most recent snapshot messages are delivered; the 5 oldest are dropped; no trade or signal message is dropped in the same scenario
  Fail if     : Fewer than 20 snapshots delivered (over-dropping) or the oldest snapshots are delivered instead of newest
  Automatable : partial
  @functional

🟡 Per-socket try/catch ensures one broken socket does not prevent broadcast to healthy sockets
  Scenario    : Open 3 WS connections; forcibly break the write stream of one (simulate ECONNRESET); call broadcastTrade; observe the other two clients
  Pass if     : The two healthy clients receive the trade message; the broken socket error is caught and logged without rethrowing
  Fail if     : The broadcast function throws and the healthy clients do not receive the message
  Automatable : partial (requires socket fault injection)
  @functional

🟡 trade-executor.ts calls broadcastTrade('trade_open') after a successful INSERT
  Scenario    : In a unit or integration test, call openTrade(); spy on broadcastTrade; assert it was called with type='trade_open' and the correct trade payload
  Pass if     : broadcastTrade spy is called exactly once with expected args
  Fail if     : Spy is never called, called with wrong type, or called before the INSERT
  Automatable : yes
  @functional

🟡 trade-manager.ts calls broadcastTrade('trade_close') after a successful UPDATE
  Scenario    : Call closeTrade(); spy on broadcastTrade; assert it was called with type='trade_close'
  Pass if     : Spy is called once with correct type and payload
  Fail if     : Spy not called or called before the UPDATE completes
  Automatable : yes
  @functional

🟢 WS endpoint is accessible at ws://localhost:3000/ws/stream when the server is running
  Scenario    : Connect a WS client to ws://127.0.0.1:3000/ws/stream
  Pass if     : Connection is established (upgrade response received)
  Fail if     : HTTP 404 or connection refused
  Automatable : yes
  @non-blocker
